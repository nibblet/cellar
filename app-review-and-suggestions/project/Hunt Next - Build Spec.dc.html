Home v2, “Try Next” & the new “Hunt Next”

A developer-facing plan to ship the redesigned Cellar home. Most of it wires up engine code that already exists; one net-new data field and one net-new surface carry the real work.

Owner  Web team Surface  /(app)/(shell)/you/cellar Est.  ~4–6 dev-days Design ref  Cellar Home — Remix (2A)

0 · What ships

The home screen is recomposed into five stacked modules. Four of them re-skin data the backend already returns; the reasoning engines are untouched. The two items that need real backend work are called out in bold.

  1. Masthead + stat strip — bottle/cigar/hunting counts from the cellar snapshot.
  2. Palate ticker — the member's dominant traits, marquee-scrolled.
  3. Tonight's Pick — the existing deterministic Daily Pour, restyled as a poster.
  4. Try Next — “reach for what you own”: the cellar branch of Find Next, plus a net-new pour-depth field.
  5. Hunt Next (new surface) — palate-matched bottles you don't own yet, with allocation-rarity stamps and a Want action.

2 · Try Next — “from your humidor”

Two cards, the palate-fit-ranked products you already have. This is the cellar branch of loadProductSuggestions — cosine similarity of each Have product's trait_vector against the per-type taste vector. No changes to the ranking.

The only new thing the card shows is pour depth (“¾ full”, “3 left”). That value does not exist today — see §4. Until the field ships, render the card without the depth chip rather than faking it.

Winston one-liner — the italic line under each product reuses the existing rationale generator (taste/rationale.ts / generateRationales). It is already cached per pick; do not call it per render.

3 · Hunt Next — the new surface

A horizontally-scrolling rail of bottles the member does not own but that fit the palate — the “chase” list. This is the catalog branch of Find Next, i.e. ensureTasteRecommendations, which already excludes everything in the cellar (have/want/tried/loved) and ranks the confirmed catalog by taste-vector cosine plus a preference boost.

3.1 Data source

  • Call loadFindNextSuggestions() and read pour + smoke, filtering to source === "catalog" (these carry suggestion_kind: "hunt_next").
  • Interleave bourbon + cigar picks into one rail, palate-score descending.
  • Raise the cap. RECOMMENDATIONS_PER_TYPE is currently 3; the rail wants ~6–8. Bump the limit passed into recommendForType for this surface only (do not change the Try Next default).

3.2 Rarity stamp (“Allocated” / “Lottery”)

The rotated wanted-poster stamp is driven by the bourbon allocation tier already produced by bourbon-tier.ts and stored on specs.tier (1–5):

  • tier 5 → “Lottery” stamp (BTAC, Pappy-class).
  • tier 4 → “Allocated” stamp.
  • tier ≤ 3 → no stamp (findable; the stamp should feel earned).
  • Cigars have no tier model — omit the stamp, or gate behind a future cigar-rarity signal.
Respect hide_rare. Members have a member_preferences.hide_rare flag and a max_catalog_tier. Hunt Next must honor both — a member who hid rare bottles should not see a rail that is nothing but lottery stamps. Filter candidates by max_catalog_tier before capping.

3.3 The “＋ Want” action

Tapping Want writes member_saves.want = true for that product via the existing cellar action (cellar/actions.ts). Two consequences to wire:

  • The product immediately leaves Hunt Next (it is now in the cellar exclude set) — optimistically remove the card, don't wait for refetch.
  • The “Hunting” count in the stat strip increments. Source that count from want-set size so the two always agree.
  • The want list itself is palate-re-ranked by rankWants() — already built, feeds the You/Want view.

4 · Net-new backend — pour depth

The one genuine data-model addition. Today member_saves is binary (have/want/tried/loved); the design shows how much is left. This is the deferred Phase 5.5 “fill level”.

-- migration: member_saves fill level
alter table cellar.member_saves
  add column fill_level smallint
  check (fill_level between 0 and 4)
  default null;
-- 4=full · 3=¾ · 2=½ · 1=heel · 0=empty · null=untracked
-- cigars: reuse as a count instead (see below)
  • Bourbon uses the 0–4 fill band → renders as the “¾ full” chip and the 4-segment meter.
  • Cigars are discrete — add a parallel count_remaining smallint (renders “3 left”), or overload one nullable quantity column and branch on product type in the loader. Pick one; don't ship both.
  • null = untracked → show no chip. Never assume “full”.
  • RLS: the new columns inherit the existing member_saves row policies — no new policy, but confirm the update policy covers the added columns.
  • Follow-on (out of scope here): an “empty” transition can auto-set tried = true and feed a restock nudge. Note it; don't build it yet.

6 · Edge & cold-start states

  • Empty cellar (no have): Try Next has nothing to rank. Collapse it and let Hunt Next lead — it works from preferences alone on a cold start (recommendForType preference-only path).
  • No palate + no preferences: both recommenders return empty by design (“we don't guess from an empty palate”). Show a single capture CTA, not empty rails.
  • Untracked fill: chip and meter hidden; never render a default level.
  • Reduced motion: the palate marquee and ember pulses must freeze — the design already guards this with @media (prefers-reduced-motion); keep it.
  • Winston on capture: this home screen may show Winston copy, but do not let the voice leak onto any capture form (per the design-system rule and the open FIX tickets).

7 · Acceptance criteria

  • Stat-strip “Hunting” count equals the want-set size at all times, including right after a Want tap.
  • A tier-5 bourbon in Hunt Next shows a “Lottery” stamp; a tier-2 shows none.
  • A member with hide_rare = true sees no tier-5 cards in Hunt Next.
  • Tapping Want removes the card without a full reload and the bottle appears in the You/Want list.
  • Setting a bourbon to “empty” shows the empty meter; an untracked bottle shows no meter.
  • Nothing in Hunt Next is already in the member's cellar (have/want/tried/loved all excluded).
  • All existing find-next, taste, and pairing unit tests still pass; new fill_level loader has coverage.

8 · Open questions for product

  1. Cigar fill: discrete count (“3 left”) vs. a box/single distinction — confirm the model before task 3.
  2. Should Hunt Next ever surface allocated bottles the member can't realistically buy, or cap at max_catalog_tier strictly? (Aspiration vs. actionability.)
  3. Does Want on an allocated bottle eventually feed a “notify me / drop tracker,” or stay a passive wishlist? (Affects whether Want needs a source/store field.)
  4. Cigar rarity: is there appetite for a cigar-side tier model so cigars can earn stamps too?
